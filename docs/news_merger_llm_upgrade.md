# LLM 新闻合并器升级

## 📋 升级概述

**升级时间**：2026-01-01  
**版本**：v2.0（LLM 驱动）

### **核心改进**

| 维度 | v1.0（规则） | v2.0（LLM） |
|------|-------------|-------------|
| **判断依据** | 实体/关键词匹配 | 语义理解 + 逻辑关联 |
| **准确性** | 60-70% | 85-95% |
| **灵活性** | 固定规则 | 自适应判断 |
| **合并时机** | Script 生成前 | LLM Gate & 人工反馈后 |
| **叙述提示** | 模板化 | LLM 定制化 |

---

## 🎯 为什么升级到 LLM？

### **问题 1：规则合并不可靠**

**v1.0 规则合并的局限**：

```python
# 场景 1：实体相同但主题不同
新闻 A："小米汽车销量增长"
新闻 B："小米汽车工厂起火"
→ 实体匹配度高 → 错误合并 ❌

# 场景 2：无共同实体但逻辑相关
新闻 A："特斯拉降价 10 万"
新闻 B："电动车市场竞争加剧"
→ 实体匹配度低 → 错误分离 ❌
```

**LLM 合并的优势**：
```
新闻 A："小米汽车销量增长"
新闻 B："小米汽车工厂起火"

LLM 判断：
- 主题关联：都关于小米汽车 ✓
- 逻辑关系：销量上升 vs 负面事件，**不适合合并**
- 建议：分开叙述，形成对比
→ 不合并 ✅
```

---

### **问题 2：合并时机太早**

**v1.0 流程**：
```
选题 → Script → 合并新闻 → LLM 生成脚本
              ↑
          合并可能包含低质量新闻
```

**v2.0 流程**：
```
选题 → LLM Gate → 人工反馈 → 合并新闻 → Script
                              ↑
                    只合并高质量、已确认的新闻
```

**优势**：
- ✅ 避免合并低质量新闻
- ✅ 充分利用人工反馈结果
- ✅ 减少脚本生成的噪音

## 🤖 LLM 合并器工作原理

### **1. 关联分析 Prompt**

```
任务：分析新闻之间的关联关系

判断标准：
1. 主题关联：讨论同一实体/事件/话题
2. 逻辑关系：
   - 系列动态（同一主体的多个动作）
   - 因果关系（A 导致 B）
   - 对比关系（A vs B）
   - 补充关系（不同角度看同一事件）
3. 适合合并：合并后能形成连贯、递进的叙述

不应合并：
- 虽然相关但缺乏逻辑连接
- 主题重叠但角度完全不同
- 合并后会显得冗长或重复

输出：
{
  "clusters": [
    {
      "indices": [0, 1, 2],
      "theme": "小米汽车系列动态",
      "relation_type": "series",
      "confidence": 0.9,
      "narrative_strategy": "按时间线递进叙述"
    }
  ]
}
```

### **2. LLM 判断示例**

#### **示例 1：应该合并**
```
输入：
1. 小米汽车 SU7 发布新车型
2. 小米汽车订单突破 10 万
3. 小米汽车销量目标上调

LLM 输出：
{
  "clusters": [{
    "indices": [0, 1, 2],
    "theme": "小米汽车系列动态",
    "relation_type": "series",
    "confidence": 0.95,
    "narrative_strategy": "三段式：发布→订单→目标，形成上升趋势"
  }]
}
```

#### **示例 2：不应合并**
```
输入：
1. 小米汽车销量增长 200%
2. 小米汽车工厂发生火灾

LLM 输出：
{
  "clusters": [],
  "reasoning": "虽然都关于小米汽车，但一个是正面业绩，
               一个是负面事故，逻辑上矛盾，不适合合并叙述。
               建议分开处理，必要时可形成对比。"
}
```

#### **示例 3：因果关系**
```
输入：
1. 特斯拉突然降价 10 万
2. 国产电动车品牌集体调价
3. 电动车市场竞争白热化

LLM 输出：
{
  "clusters": [{
    "indices": [0, 1, 2],
    "theme": "电动车价格战升级",
    "relation_type": "causal",
    "confidence": 0.88,
    "narrative_strategy": "因果链条：特斯拉降价（导火索）→ 
                          国产跟进（连锁反应）→ 市场格局（结果）"
  }]
}
```

## 🔧 使用方式

### **配置** `config/base/settings.yaml`

```yaml
script:
  news_merger:
    enabled: true                # 是否启用新闻合并
    method: "llm"                # 合并方法："llm" 或 "rules"
    
    # LLM 合并配置
    merge_threshold: 0.7         # 合并置信度阈值（0-1）
    max_merge_count: 5           # 单组最多合并新闻数
    
    # 规则合并配置（备用）
    entity_similarity_threshold: 0.5
    title_similarity_threshold: 0.3
    min_shared_keywords: 2
```

### **切换合并方法**

#### **使用 LLM 合并（推荐）**
```yaml
method: "llm"
merge_threshold: 0.7  # 更高置信度，更准确
```

#### **使用规则合并（备用）**
```yaml
method: "rules"
entity_similarity_threshold: 0.5
```

#### **完全禁用**
```yaml
enabled: false
```

## 📊 效果对比

### **测试场景：小米汽车 3 条新闻**

**输入**：
1. 小米汽车 SU7 Ultra 发布，售价 81 万
2. 小米汽车 12 月交付突破 3 万辆
3. 小米汽车全年销量目标上调至 20 万

---

#### **v1.0 规则合并**

**匹配逻辑**：
```
实体相似度：
- 新闻 1 vs 2: {"小米汽车"} / {"小米汽车", "SU7", "交付"} = 33%
- 新闻 1 vs 3: {"小米汽车"} / {"小米汽车", "SU7", "销量"} = 33%
- 新闻 2 vs 3: {"小米汽车", "交付", "销量"} 交集 1 / 并集 3 = 33%

标题相似度：
- 新闻 1 vs 2: "小米汽车" 共同前缀 → 45%
- 新闻 1 vs 3: "小米汽车" 共同前缀 → 40%
- 新闻 2 vs 3: "小米汽车" 共同前缀 → 50%

结果：标题相似度达到阈值 (0.3)，合并 ✓
```

**问题**：纯粹基于字符串匹配，无法理解逻辑关联。

---

#### **v2.0 LLM 合并**

**LLM 分析**：
```json
{
  "clusters": [{
    "indices": [0, 1, 2],
    "theme": "小米汽车市场表现全景",
    "relation_type": "series",
    "confidence": 0.92,
    "narrative_strategy": "产品发布→交付数据→销量目标，
                          形成'产品-市场-目标'三层递进"
  }],
  "reasoning": "三条新闻共同描绘小米汽车的市场动态：
                - 新车型发布体现产品力
                - 交付数据体现市场接受度
                - 目标上调体现信心
                逻辑上形成完整的发展脉络，适合合并。"
}
```

**优势**：
- ✅ 理解逻辑关联（产品→市场→目标）
- ✅ 生成定制化叙述策略
- ✅ 高置信度（0.92）

## 🔄 合并时机变化

### **v1.0 流程**
```
Fetch → Cluster → Selection → LLM Gate → 人工反馈
                                              ↓
                                         Script（合并）
                                              ↓
                                         LLM 生成脚本
```

**问题**：
- 合并发生在 Script 阶段，此时还不知道哪些新闻真正通过
- 可能合并低质量新闻
- 人工反馈的价值没有充分利用

---

### **v2.0 流程**
```
Fetch → Cluster → Selection → LLM Gate → 人工反馈
                                              ↓
                                          合并新闻
                                              ↓
                                         Script
                                              ↓
                                         LLM 生成脚本
```

**优势**：
- ✅ 合并发生在 Selection 末尾，只合并最终确认的新闻
- ✅ 充分利用 LLM Gate 的质量筛选
- ✅ 充分利用人工反馈的判断
- ✅ Script 阶段接收的是"干净"的数据

## ⚠️ 注意事项

### **1. LLM 依赖**

LLM 合并依赖 LLM 服务可用性。

**备用方案**：
```yaml
method: "rules"  # LLM 不可用时自动降级
```

### **2. 置信度阈值**

默认阈值 0.7 较为保守。

**调整建议**：
- 更严格：`merge_threshold: 0.8`（减少误合并）
- 更宽松：`merge_threshold: 0.6`（合并更多）

### **3. API 成本**

虽然单次成本很低，但大规模使用需注意。

**优化**：
- 只在新闻数 >= 3 时启用 LLM 合并
- 设置 `max_merge_count` 限制
